# Phase 8: Retrospect

## 권한 선언

| 허용 | 금지 |
|------|------|
| 모든 아티팩트 파일 읽기 | 프로젝트 소스 파일 수정 |
| `${CLAUDE_SKILL_DIR}/memory.md` 쓰기 | git 명령 |
| 스킬 파일 읽기 | 스킬 파일 직접 수정 |

이 페이즈에서 위 금지 항목이 필요하다고 판단되면 즉시 중단하고 사용자에게 알린다.

## 목적

이번 harness 실행을 분석하여 `${CLAUDE_SKILL_DIR}/memory.md`를 업데이트하고,
harness 스킬 구조 문제에서 비롯된 반복 실패가 있으면 개선을 제안한다.

## PRE-EVENT

### 사전 조건 확인
- `$HARNESS_DIR/evaluate.md` 존재 확인 → 없으면 "회고할 실행 아티팩트가 없습니다" 출력 후 종료
- `$HARNESS_DIR/clarify.md`에서 프로젝트명 추출 → memory.md 섹션 키로 사용

프로젝트명 추출 규칙:
1. `$HARNESS_DIR/clarify.md`의 "요청 요약" 첫 줄에서 추출
2. 없으면 CLAUDE.md의 프로젝트 이름 사용
3. 없으면 현재 디렉토리명 사용

## 절차

### Step 1: 아티팩트 읽기

아래 파일을 순서대로 읽는다:

1. `$HARNESS_DIR/clarify.md` — 작업 유형, 성공 기준 파악
2. `$HARNESS_DIR/evaluate.md` — 성공 기준별 PASS/FAIL 결과
3. `$HARNESS_DIR/test.md` — 테스트 실행 결과 (실패 여부)
4. `$HARNESS_DIR/generate.md` — 변경 파일 목록, wave 실행 이력

### Step 2: 패턴 분석

아래 항목을 분석한다:

**반복 실패 패턴 후보:**
- evaluate.md에 FAIL 항목이 있는가?
- generate.md → test.md → generate.md 재실행 흔적이 있는가? (generate.md의 "태스크별 결과"에 재실행 언급 여부)
- 특정 파일 확장자(.py, .js, .sql 등)에서 문제가 집중되었는가?

**효과적 패턴 후보:**
- Wave 분리가 충돌 없이 성공했는가?
- 병렬 실행이 순차 실행 대비 명확히 유리했는가?

**프로젝트 고유 주의사항 후보:**
- 이번 작업에서 처음 발견된 환경/도구/의존성 관련 특이사항

**스킬 개선 제안 후보:**
- 반복 실패 원인이 harness 스킬 구조(페이즈 설계, 권한 설정, handoff 형식 등)에 있는가?
- 프로젝트 코드 문제나 사용자 실수이면 스킬 제안 대상이 아님

### Step 3: memory.md 업데이트

`${CLAUDE_SKILL_DIR}/memory.md`를 읽는다.

**프로젝트 섹션이 없으면:** 파일 끝에 아래 섹션을 추가한다:

```
## [{프로젝트명}] 반복 실패 패턴
| 유형 | 발생 횟수 | 마지막 발생 | 권고 대응 |
|------|-----------|-------------|-----------|

## [{프로젝트명}] 효과적이었던 패턴

## [{프로젝트명}] 프로젝트 고유 주의사항
```

**반복 실패 패턴 업데이트:**
- 분석에서 발견된 실패 유형이 테이블에 이미 있으면 → 발생 횟수 +1, 마지막 발생 날짜 갱신
- 없으면 → 신규 행 추가 (발생 횟수: 1, 마지막 발생: 오늘, 권고 대응: 분석 결과 기반 작성)

**효과적이었던 패턴 업데이트:**
- 이번 실행에서 특히 잘 작동한 것만 추가
- 이미 동일한 내용이 있으면 추가하지 않는다

**프로젝트 고유 주의사항 업데이트:**
- 이번에 새로 발견된 것만 추가 (형식: `- {내용} ({오늘 날짜})`)
- 이미 동일한 내용이 있으면 추가하지 않는다

### Step 4: 스킬 개선 제안 (선택)

반복 실패 패턴이 harness 스킬 구조 문제에서 비롯된 경우에만 실행한다.

`${CLAUDE_SKILL_DIR}/memory.md`의 `## 스킬 개선 제안` 섹션에 추가:

```
- [ ] {구체적 제안 내용} — 근거: {이번 실행에서 관찰된 내용 1줄}
```

### Step 5: 승인 항목 정리

`## 스킬 개선 제안` 섹션에서 `- [x]`로 표시된 항목 수를 센다.

- 5건 이하: 건드리지 않는다
- 6건 이상: 위에서부터(가장 오래된 순) 초과분만 삭제한다
  - 예: 7건이면 위에서 2건 삭제

### Step 6: 사용자에게 요약 출력

> "memory.md 업데이트 완료 (`${CLAUDE_SKILL_DIR}/memory.md`):
>  - [{프로젝트명}] 반복 실패 패턴: {N}건 업데이트
>  - [{프로젝트명}] 효과적 패턴: {N}건 추가
>  - [{프로젝트명}] 프로젝트 고유 주의사항: {N}건 추가
>  - 스킬 개선 제안: {N}건 추가 (미승인)
>
>  개선 제안을 적용하려면 memory.md를 확인 후 해당 항목을 `[x]`로 변경해주세요."

## POST-EVENT

### 구조 검증 (LLM 없이)
- `${CLAUDE_SKILL_DIR}/memory.md` 파일이 존재하는지 확인
- `## 스킬 개선 제안` 섹션이 존재하는지 확인
- `[x]` 항목이 5건 이하인지 확인

## 완료 조건

memory.md 업데이트 완료 후 종료한다. Phase 8은 마지막 페이즈이므로 다음 페이즈가 없다.
