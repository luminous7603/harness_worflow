# Phase 3: Plan

## 권한 선언

| 허용 | 금지 |
|------|------|
| 프로젝트 파일 읽기 | 프로젝트 파일 수정/생성/삭제 |
| `$HARNESS_DIR/plan.md` 쓰기 | 그 외 파일 쓰기 |
| `.harness/tasks.json`, `.harness/phases.json` 쓰기 | Bash 명령 실행 |

이 페이즈에서 위 금지 항목이 필요하다고 판단되면 즉시 중단하고 사용자에게 알린다.

## PRE-EVENT

### 사전 조건 확인
- `$HARNESS_DIR/clarify.md` 존재 확인 → 없으면 Phase 1 재실행 요청
- `$HARNESS_DIR/context.md` 존재 확인 → 없으면 Phase 2 재실행 요청

### 목표 리마인더
`$HARNESS_DIR/clarify.md`에서 다음을 추출하여 작업 전에 확인한다:

> **이 작업의 목표:** {요청 요약 1줄}  
> **성공 기준:** {성공 기준 항목들}  
> **핵심 제약:** {제약 조건 항목들}

이 목표와 무관한 태스크는 계획에 포함하지 않는다.

### Memory 참고

`${CLAUDE_SKILL_DIR}/memory.md`가 존재하면 읽는다. 존재하지 않으면 이 단계를 건너뛴다.

현재 프로젝트의 `## [{프로젝트명}] 반복 실패 패턴` 테이블에서 **발생 횟수가 2회 이상**인 항목을 찾는다.

해당 항목이 있으면:
- `plan.md`에 태스크를 작성할 때 해당 유형과 관련된 태스크 설명에 "⚠️ 반복 실패 주의: {권고 대응}" 을 명시한다
- 해당 태스크를 다른 태스크와 병렬 배치하지 않고 단독 Wave로 분리한다

## 목적

Clarify와 Context Gather 결과를 바탕으로 구체적인 실행 계획을 수립한다.

## 절차

1. `$HARNESS_DIR/clarify.md`와 `$HARNESS_DIR/context.md`를 읽는다
2. 구현에 필요한 태스크 목록을 작성한다
3. 각 태스크에 대해 의존성을 분석한다:
   - **병렬 가능:** 서로 다른 파일을 수정하고 의존성이 없는 태스크
   - **순차 필수:** 이전 태스크 결과에 의존하는 태스크
4. 각 태스크에 대해 수정/생성할 파일 목록을 명시한다
5. **범위 검토**를 수행한다:
   - 기존 코드로 해결 가능한 부분이 있는지 확인 (`$HARNESS_DIR/context.md` 참조) — 동일한 기능을 수행하는 함수/클래스가 이미 존재하거나 80% 이상 로직이 중복되는 경우 재사용 대상으로 판단한다
   - 이번에 구현하지 않는 항목을 명시한다
   - 신규 파일이 5개를 초과하면 ⚠️ 복잡도 경고를 plan.md에 기재하고 사용자에게 분할 여부를 질문한 뒤 승인을 받아 진행한다
6. **실패 시나리오**를 작성한다:
   - 각 태스크에서 발생 가능한 주요 실패 경로를 최소 1개, 최대 2개 식별한다
   - 데이터 변경/삭제/외부 API 호출이 포함된 태스크는 해당 태스크에 대해 실패 시나리오를 반드시 1개 이상 포함한다
7. 아티팩트로 저장한다
8. `.harness/phases.json`과 `.harness/tasks.json`을 생성 또는 갱신한다 (`utils/state-helpers.md` 참조)

## 아티팩트 저장

`$HARNESS_DIR/plan.md`에 아래 형식으로 저장한다:

```markdown
---
phase: plan
status: in-review
timestamp: {현재 ISO 8601}
next: generate
---

# Plan

## 태스크 목록

### [순차] Task 1: {태스크명}
- 수정 파일: {파일 경로}
- 생성 파일: {파일 경로}
- 설명: {무엇을 구현하는가}

### [병렬] Task 2: {태스크명}
- 수정 파일: {파일 경로}
- 설명: {무엇을 구현하는가}

### [병렬] Task 3: {태스크명}
- 수정 파일: {파일 경로}
- 설명: {무엇을 구현하는가}

## 실행 순서
1. Task 1 (순차)
2. Task 2 + Task 3 (병렬)

## 범위 검토
- 기존 코드 재사용 가능 부분: {있으면 파일명과 함께 명시, 없으면 "없음"}
- 이번에 하지 않는 것: {명시. 없으면 "없음"}
- 복잡도: {신규 파일 수 N개 — 5개 초과 시 ⚠️ 경고, 사용자 승인 후 진행}

## 실패 시나리오
| 태스크 | 실패 경로 | 대응 |
|---|---|---|
| {Task명} | {실패 조건} | {대응 방법} |
```

## POST-EVENT

### 구조 검증 (LLM 없이)
- `$HARNESS_DIR/plan.md` 생성 확인
- `.harness/tasks.json`에 최소 1개 태스크가 정의되었는지 확인
- 각 태스크에 `files.modify` 또는 `files.create`가 명시되었는지 확인
- 동일 파일 경로를 수정하는 태스크가 2개 이상이면 의존성 재검토 요청
- "범위 검토" 섹션이 존재하는지 확인
- "실패 시나리오" 섹션에 최소 1개 행이 있는지 확인

### Handoff 생성
`$HARNESS_DIR/handoff-03to04.md`를 아래 내용으로 생성한다:

```markdown
# Handoff: Phase 3 → Phase 4

## 목표 (Goal)
{clarify.md의 요청 요약 1줄}
성공 기준: {성공 기준 목록}

## 이번 페이즈가 파악한 것 (Findings)
- 태스크 목록: {태스크명 + 담당 파일}
- Wave 분류 결과: Wave 1 = {태스크들}, Wave 2 = {태스크들}

## 다음 페이즈가 해야 할 것 (Next)
- Wave 1 병렬 실행: {태스크 목록}
- Wave 2 이후 순서: {태스크 목록}

## 참고 파일 (References)
- `$HARNESS_DIR/plan.md` (상세 태스크 설명 필요 시)
```

## 승인 게이트

계획 작성 완료 후 사용자에게 계획을 보여주고 승인을 요청한다:

> "Phase 3 Plan이 완료되었습니다. 위 계획을 검토해주세요. 진행할까요?"

사용자가 승인하면 `plan.md`의 frontmatter `status`를 `approved`로, `timestamp`를 현재 시각으로 갱신한 후 Phase 4로 진행한다. 수정 요청이 있으면 반영 후 재승인을 요청한다.
