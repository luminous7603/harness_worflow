# Phase 2: Context Gather

## 권한 선언

| 허용 | 금지 |
|------|------|
| 프로젝트 파일 읽기 | 프로젝트 파일 수정/생성/삭제 |
| `$HARNESS_DIR/context.md` 쓰기 | 그 외 파일 쓰기 |
| `.claude/.harness/context.md` 쓰기 (최초 1회) | Bash 실행 (읽기 전용 명령 제외) |

읽기 전용 허용 Bash: `find`, `ls`, `cat`, `glob` 계열만 허용.  
이 페이즈에서 위 금지 항목이 필요하다고 판단되면 즉시 중단하고 사용자에게 알린다.

## PRE-EVENT

### 사전 조건 확인
- `$HARNESS_DIR/clarify.md` 존재 확인 → 없으면 Phase 1 재실행 요청

### Memory 로드

`${CLAUDE_SKILL_DIR}/memory.md`가 존재하는지 확인한다:

- **존재하지 않으면:** 건너뛴다 (첫 실행 또는 memory 미사용)
- **존재하면:**
  1. 파일을 읽는다
  2. 현재 프로젝트명에 해당하는 섹션을 추출한다
     - 프로젝트명은 `$HARNESS_DIR/clarify.md`의 요청 요약에서 추출하거나, CLAUDE.md의 프로젝트명을 사용한다
  3. 해당 섹션의 내용을 아래 형식으로 기억해두어 서브에이전트 프롬프트에 주입한다:

```
## 과거 학습 (참고)
### 반복 실패 패턴 (주의)
{memory.md에서 추출한 해당 프로젝트의 반복 실패 패턴 테이블 — 없으면 이 항목 생략}

### 프로젝트 고유 주의사항
{memory.md에서 추출한 해당 프로젝트의 고유 주의사항 — 없으면 이 항목 생략}
```

이 블록을 서브에이전트 프롬프트 템플릿의 "**반드시 포함할 출력:**" 섹션 바로 앞에 추가한다.

### LESSONS_LEARNED.md 확인 (선행 작업)

`skills/harness/LESSONS_LEARNED.md`가 존재하면 읽고, 이번 프로젝트명과 관련된 항목을 찾는다.
파일이 없으면 건너뛴다. 오류를 내지 마세요.

관련 항목이 있으면 서브에이전트 프롬프트 템플릿의 `## 반드시 포함할 출력:` 섹션 앞에 아래 블록을 추가한다:

```
## 과거 실패 패턴 주의 (LESSONS_LEARNED)
{관련 항목 내용 — "다음 실행 시 주의사항" 항목만 발췌}
```

### 목표 리마인더
`$HARNESS_DIR/clarify.md`에서 다음을 추출하여 작업 전에 확인한다:

> **이 작업의 목표:** {요청 요약 1줄}  
> **성공 기준:** {성공 기준 항목들}  
> **핵심 제약:** {제약 조건 항목들}

이 목표와 무관한 파일 탐색은 수행하지 않는다.

## 목적

코드베이스의 구조와 컨벤션을 파악한다.

## 절차

Phase 2는 서브에이전트에 위임하여 메인 컨텍스트를 보호한다.  
단, `.claude/.harness/context.md`가 이미 존재하면 서브에이전트를 호출하지 않고 Step 0만 수행한다.

### Step 0: 프로젝트 레벨 context.md 확인

`.claude/.harness/context.md`가 존재하는지 확인한다:

- **존재하면:** 파일을 읽고 내용을 기반으로 추가 탐색을 최소화한다. 현재 요청과 다른 점이 있으면 보완만 한다. 아래 서브에이전트 호출은 생략한다.
- **존재하지 않으면:** 아래 서브에이전트 호출을 수행한다.

### 서브에이전트 호출 (`.claude/.harness/context.md` 미존재 시)

Agent 도구를 호출하여 아래 프롬프트로 탐색을 위임한다:

---
**[서브에이전트 프롬프트 템플릿]**

당신은 코드베이스 탐색 전문 에이전트입니다.

**목표:** {handoff-01to02.md의 Goal 섹션}

**탐색 범위:** {handoff-01to02.md의 Next 섹션의 키워드/영역}

**수행할 작업:**
1. 프로젝트 루트 파일 구조 파악 (Glob)
2. CLAUDE.md / README.md / RULES.md 읽기
3. 목표와 관련된 기존 코드 파일 읽기
4. 코드 컨벤션 패턴 파악 (네이밍, 구조, 임포트, 에러 처리)

**반드시 포함할 출력:**
```markdown
# Context

## 프로젝트 구조 요약
{주요 디렉토리 및 역할}

## 코드 컨벤션
{네이밍, 구조, 패턴 — 3줄 이내 핵심만}

## 관련 기존 파일
{요청과 관련된 파일 목록 및 역할}

## 참고 사항
{구현 시 주의할 사항}
```

---

### 서브에이전트 완료 후

1. 반환된 context 내용을 `$HARNESS_DIR/context.md`에 저장한다 (frontmatter 포함)
2. `.claude/.harness/context.md`가 없으면 프로젝트 전반 정보를 포함하여 생성한다
3. POST-EVENT 절차를 수행한다

## POST-EVENT

### 구조 검증 (LLM 없이)
- `$HARNESS_DIR/context.md` 파일이 생성되었는지 확인
- "관련 기존 파일" 섹션에 최소 1개 파일이 명시되었는지 확인

### Handoff 생성
`$HARNESS_DIR/handoff-02to03.md`를 아래 내용으로 생성한다:

```markdown
# Handoff: Phase 2 → Phase 3

## 목표 (Goal)
{clarify.md의 요청 요약 1줄}
성공 기준: {성공 기준 목록}

## 이번 페이즈가 파악한 것 (Findings)
- 관련 파일 목록: {파일명 + 역할 1줄}
- 코드 컨벤션 핵심: {3줄 이내}

## 다음 페이즈가 해야 할 것 (Next)
- Plan에서 반드시 고려해야 할 제약 사항: {제약 목록}

## 참고 파일 (References)
- `$HARNESS_DIR/context.md` (상세 컨벤션 필요 시)
- `.claude/.harness/context.md` (프로젝트 전반 정보)
```
