# Phase 1: Clarify

## 권한 선언

| 허용 | 금지 |
|------|------|
| 프로젝트 파일 읽기 | 프로젝트 파일 수정/생성/삭제 |
| `$HARNESS_DIR/clarify.md` 쓰기 | 그 외 파일 쓰기 |
| 사용자와 대화 | Bash 명령 실행 |

이 페이즈에서 위 금지 항목이 필요하다고 판단되면 즉시 중단하고 사용자에게 알린다.

## PRE-EVENT

- (없음) 첫 페이즈이므로 사전 조건 없음

## 목적

요구사항을 명확히 하고 성공 기준을 정의한다.

## 절차

1. 사용자의 요청을 읽는다
2. 요청이 너무 크거나 여러 독립 시스템에 걸쳐 있으면, 서브태스크로 분해할 것을 제안한다
3. 아래 항목을 파악할 때까지 질문한다 (한 번에 하나씩):
   - 구현 목적
   - 제약 조건 (기술, 환경, 기간 등)
   - 성공 기준 (어떻게 되면 완료인가?)
4. **전제 확인** — 아래 두 가지를 검토하여 해당하는 경우에만 사용자에게 질문한다:
   - 기존 코드가 이 문제를 이미 부분적으로 해결하고 있는가? (사용자 메시지에 파일명·모듈명·기존 기능이 언급된 경우에 한해 검토)
   - 이 구현 없이도 목적을 달성할 더 단순한 방법이 있는가?
   위 두 항목 중 해당하는 것이 없거나 판단 근거가 없으면 이 단계를 건너뛴다.
5. **대안 접근법 제시** — 3번 완료 후에도 아래 조건 중 하나라도 해당하면 최소 2가지 접근법을 제시하고 사용자가 선택하도록 한다:
   - 성공 기준은 확정됐으나 구현 방향이 여전히 여러 갈래인 경우
   - 요청 범위가 커서 "전부 구현"과 "핵심만 구현" 선택이 필요한 경우
   접근법 형식: `[접근법명] — {한 줄 설명} / 예상 범위: {작음/중간/큼}`
   조건에 해당하지 않으면 이 단계를 건너뛴다.
6. 파악된 내용을 아티팩트로 저장한다

## 아티팩트 저장

`utils/artifact-helpers.md`를 참고하여 `$HARNESS_DIR/clarify.md`에 아래 형식으로 저장한다:

```markdown
---
phase: clarify
status: approved
timestamp: {현재 ISO 8601}
next: context-gather
---

# Clarify

## 요청 요약
{사용자 요청 한 줄 요약}

## 목적
{구현 목적}

## 제약 조건
{제약 조건 목록}

## 성공 기준
{완료 기준 목록}

## 선택한 접근법
{대안 제시가 있었던 경우 선택 결과 기록. 없었으면 "해당 없음"}
```

## POST-EVENT

### 구조 검증 (LLM 없이)
- `$HARNESS_DIR/clarify.md` 파일이 생성되었는지 확인
- "성공 기준" 섹션이 존재하고 1개 이상의 항목이 있는지 확인
- "선택한 접근법" 섹션의 내용이 공백이 아닌지 확인 ("해당 없음" 포함)
- 비어 있으면: 사용자에게 성공 기준 재확인 요청

### Handoff 생성
`$HARNESS_DIR/handoff-01to02.md`를 아래 내용으로 생성한다:

```markdown
# Handoff: Phase 1 → Phase 2

## 목표 (Goal)
{clarify.md의 요청 요약 1줄}
성공 기준: {성공 기준 목록}

## 이번 페이즈가 파악한 것 (Findings)
(없음 — Phase 1은 첫 페이즈)

## 다음 페이즈가 해야 할 것 (Next)
- 다음 키워드/영역을 중심으로 탐색: {요청 키워드 기반 탐색 영역}

## 참고 파일 (References)
(없음)
```

## 완료 조건

성공 기준이 명확히 정의되면 Phase 2로 진행한다.

완료 후 "DONE"을 보고하세요.
